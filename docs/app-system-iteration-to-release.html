V10 迭代视角

系统层：App 迭代 → 迭代发布完成

本文讨论的是：已有 live app 被继续编辑时，系统层如何处理 linked draft、Candidate、发布验证以及 promote。

1. 当前推荐的迭代形态

  • 已有 live app 不应直接编辑上线版本本体。
  • 应从 live 派生出 linked workspace draft。
  • draft 是升级工作区，live 是线上运行实体。
  • draft 在验证成功前，不应影响 live 运行。

2. 迭代系统流程

Live app → /ensure-workspace-draft 创建/获取 linked draft → 在 draft 中开发与保存 → Draft 发起 Candidate 发布 → Candidate 验证 → 通过后 promote 到 live → draft 进入已完成/归档语义

3. 关键系统点

3.1 linked draft 创建

  • 入口：/api/apps/:id/ensure-workspace-draft
  • 要求来源必须是 live release。
  • draft 会携带 release_app_id 指向原 live app。

3.2 draft 工作期

  • draft 内继续聊天开发、保存版本、预览、保存 draft。
  • 此阶段本质上是“升级工作区”。

3.3 迭代发布

  • 发布入口仍走统一 publish pipeline。
  • 但对于 workspace draft，当前版本已经开始改成：先在 draft runtime 上进行 candidate 验证，再 promote 到 live。

4. 为什么必须隔离 draft 与 live

  • 如果 draft 发布失败直接影响 live runtime，就等于把升级工作区和线上系统绑死。
  • 正确做法是：draft 先自证可运行，再切换 live。
  • 这也是 V10 升级发布模型成立的前提。

当前代码已经做过一轮关键修复，把 candidate runtime / prod DB 操作尽量先落到 draft 本身，再在 verify 成功后 promote。

5. 当前版本下的 promote 时序

  • draft 产生产物（frontend/backend/sql）
  • 启动 candidate runtime
  • 应用 schema / 做 runtime health
  • 做 verifier
  • 通过后把版本产物写入 release app
  • 部署 release runtime
  • 最终校验后标记 live

6. 迭代完成后的状态理解

  • 对用户来说：draft 可以“消失”或隐藏。
  • 对系统来说：更合理的是进入 merged / archived 语义，而不是真删除。
  • 这有助于追踪本轮升级是如何形成的。
