当前代码现状说明

系统层：App 开发 → 上线 → 上线后沙盒运行构架

本文描述的是当前 funfo AI Store 这套系统在“创建新 app、开发、发布到 live、以及上线后运行”时的系统层构架逻辑。

1. 实体层

1.1 App 实体

  • apps 表记录 app 主实体。
  • 核心字段包括：id、status、release_state、app_role、release_app_id、current_version、live_version_id、candidate_version_id。
  • 当前系统同时承载了：draft、candidate、live、failed、rollback 等状态。

1.2 版本实体

  • app_versions 记录每一次保存或发布产物。
  • 每个版本可包含三类核心产物：code（前端 JSX）、server_code（后端）、sql_code（SQL/schema）。
  • 文件系统中也会 materialize 到 server/apps/<appId>/ 下面。

2. 创建与开发期运行逻辑

用户在 Workspace 发起需求 → /api/apps/:id/chat → 生成前端 JSX / 可选后端 / 可选 SQL → 写入 app_versions → materialize 文件 → 启动 preview → 若为 server 模式则部署 backend
  • 主入口是 server/index.js 的 /api/apps/:id/chat。
  • 生成完成后会立即保存一条新版本。
  • 如果 runtime mode 是 server，则会尝试部署 backend 并启动预览。
  • 预览通过 preview_slug 或 preview 端口访问。

3. 上线（Draft → Candidate → Live）系统流程

Draft / Live 升级入口 → 发布流水线 processPublishJob() → 生成 Candidate 产物 → 启动 Candidate runtime → 数据库 schema 应用 → verifier 校验 → promote 成 live
  • 发布核心在 server/publish/pipeline.js 的 processPublishJob()。
  • 发布主步骤目前收敛为 4 步：生成 Candidate、启动并检查 Candidate 环境、发布验证、发布完成。
  • 系统会生成 frontend/backend/sql 组合产物，并进行 dry-run、runtime health、verifier 多层校验。
  • 只有通过 verify 之后，才会把 candidate promote 到 live。

4. 上线后沙盒 / 运行时结构

4.1 运行对象

  • 每个运行中的 app 会有自己的 runtime/container 语义。
  • 系统会追踪 backend、frontend、preview、health 状态。
  • Admin 面板可查看 runtime 状态，并可 wake / sleep。

4.2 数据文件

  • 运行期数据库主要围绕 SQLite 文件。
  • prod 路径会落到 app 对应目录的数据文件中。
  • 发布前后会结合 backup snapshot、schema apply、verifier 做安全检查。

5. 当前版本的重要现实状态

  • 已实现 发布不再只是简单替换，而是有 Candidate / Verify / Promote 主干。
  • 已实现 文档基线逐步收敛到 docs/。
  • 刚做关键修复 从 live 派生 draft 的发布，已开始从“直接碰 live runtime”改成“先在 draft runtime 验证，再 promote”。
  • 仍需持续验证 draft/live 隔离是系统级风险点，建议继续做真实发布演练验证。
