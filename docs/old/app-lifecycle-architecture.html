funfo：app 生成 → 发布 → 长期化（Full Docker）架构图

目标状态：长期化 app 的前端与后端都运行在同一个独立 Docker 容器中；主平台只负责网关、编排、状态管理。

阶段 1 · 开发 / 生成
Workspace 开发
用户在 funfo AI Store 中通过聊天生成 app，持续编辑前端 JSX、server_code、schema.sql。
生成产物
写入 apps、app_versions、messages，以及 server/apps/<appId> 的运行时文件。
  • 前端代码：App.jsx / code
  • 后端代码：server.js / server_code
  • 数据库：schema.sql / sqlite
  • 开发态预览：preview-manager
→
阶段 2 · 发布
发布流水线
前端分析、后端/SQL 生成、DB 检查、容器启动、健康检查、验证、完成。
发布目标
把最新版本固化为长期化 release app，并生成完整的容器运行目录。
  • 生成完整 frontend runtime HTML
  • 复制 runtime-assets（react/recharts 等）
  • 写入 server.js + index.html + schema.sql
  • 为长期化 app 创建独立容器
→
阶段 3 · 长期化 / 线上运行
Full Docker Runtime
一个长期化 app = 一个容器，容器里同时提供前端页面、/api 接口和 sqlite 数据库。
主平台职责
主平台不再托管长期化 app 的前端预览，只做 /app/<slug> 的网关代理、唤醒、休眠、运行状态管理。
  • /app/<slug>/ → 代理到 app 容器前端
  • /app/<slug>/api/* → 同容器 API
  • SQLite 使用 data_prod.sqlite
  • Admin / My Apps 可查看容器运行状态
开发态
继续允许使用 preview-manager，适合迭代、调试、快速修复。
发布态
发布阶段负责把“开发态资产”转成“长期化容器资产”。
长期态
长期化 app 必须完整 Docker 化，避免现在这种 backend-only Docker 的混合架构。
一句话总结： 长期化 app 的正确目标不是“后端在 Docker，前端靠主平台 preview”，而是 前后端都在同一个独立 app 容器中运行。
