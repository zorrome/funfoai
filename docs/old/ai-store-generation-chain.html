funfo_AI_Store 业务链路 提示词限制

AI Store 生成 App 的完整业务逻辑与链条

这份文档是基于当前 funfo_AI_Store 代码整理出来的“真实链路图”。它覆盖：首次生成、首次部署、迭代开发、迭代部署、回滚，并把每个环节会影响生成能力/结构的限制提示词和守护逻辑一起标出来。

先说结论： 当前系统不是“完全自由生成器”，而是一个带强约束的工作流生成器：开发态偏前端优先，迭代态偏最小改动，部署态会补后端并做 DB/运行时校验，发布前还会对 release 做备份。
但我没有在当前代码里看到一个完整的“用户可直接点击回滚到某个 release backup”的正式 API/UI；现在更像是已做备份能力，但未完整产品化回滚入口。

0. 总体结构图

创建 App → POST /api/apps 创建一条 app 记录，初始没有版本、没有 preview、没有 backend。
在 Workspace 中对话生成 → POST /api/apps/:id/chat 调 OpenClaw 流式生成，先产出 JSX；必要时复用旧后端或保存版本。
保存工作版本 → 新增 app_versions，落盘前端/后端/SQL 文件，启动 preview，必要时启动 backend。
部署/发布 → POST /api/apps/:id/publish 异步发布，生成或复用 production backend，校验 DB，启动 Docker runtime，写入 release 新版本。
编辑已发布 App → 不能直接改 release，先 ensure-workspace-draft 建一个 linked workspace draft，再继续开发。
保存草稿 / 部署覆盖 → draft 可以保持本地工作态；也可以 deploy 覆盖 release。覆盖前会做 release backup。

1. 首次生成（First Generation）

业务链条

入口
前端先调用 POST /api/apps 创建 app。初始 app 只有基础信息（name/icon/description），还没有版本。
然后在 Workspace 中输入需求，前端调用 api.chat(appId, text)。
发送生成请求
后端进入 POST /api/apps/:id/chat。若这是首次生成，通常没有历史版本/原始 JSX，因此“基于旧版本最小修改”的限制会弱很多。
模型输出
生成结果会被 parseAIResponse 解析成 jsx / server / sql。首次生成核心目标仍然是先得到可运行的前端 JSX。
保存版本
系统会新增一条 app_versions 记录，写入版本文件，启动 preview；如果有可用 backend，也可能同时启动 backend runtime。

首次生成阶段的限制

限制影响
DEV_FRONTEND_FIRST_MODE只要 app 还处在开发态（runtime_mode === local 或 status !== published），就会偏向“先生成前端 JSX，不先生成 backend / SQL”。
流式输出要求要求模型尽快开始输出 ```jsx 代码块，方便编辑器边生成边显示。
Phase-A 校验即使是首次生成，也要先过一轮前端可运行性校验；失败就不进入预览。
设计 token 重试如果 prompt 中有指定风格 token，生成后还会再检查一次，不满足就 restyle retry。

首次生成时最关键的限制提示词

[DEV_FRONTEND_FIRST_MODE]
Development phase is frontend-first. Generate frontend JSX only.
Do NOT generate backend routes or SQL in this phase.
When streaming, start outputting the main App code immediately inside a ```jsx block
so the editor can show code being written live.

触发条件：runtime_mode === 'local' 或 app 不是 published。首次生成几乎天然命中。

2. 首次部署（First Deploy）

业务链条

入口
前端点击 Deploy / 发布，调用 POST /api/apps/:id/publish。
异步发布任务
后端把 app 的 publish_status 标为 publishing，创建 publish job，再异步执行 processPublishJob()。
检查前端
先确认当前最新版本里有 frontend code；如果没有，会直接报错：公開前にフロントエンドを生成してください。
生成/复用 backend
首次部署通常没有可复用的已发布 backend，因此更可能进入“为发布生成 production backend”的路径。
DB 检查
执行 validatePublishSchemaSafety()，在临时 SQLite 上尝试应用 schema，检查生产 DB 相容性。
Docker 启动
调用 deployAppBackend()，启动容器/服务；随后等待 health check。
写入 release 版本
发布成功后，release app 新增一条 publish vN 版本，状态设为 published、runtime_mode=server。

首次部署阶段的限制

限制影响
必须先有前端没有 frontend code，不能部署。
生产 backend prompt部署时会重新针对前端生成 production backend，而不是完全相信开发态结果。
API contract 对齐要求 backend 跟 frontend fetch 使用的接口保持一致。
DB schema safety发布前必须过 schema 安全检查。
运行时健康检查Docker 和 preview 都要实际可达，否则发布失败。

首次部署时最关键的限制提示词

Generate production backend for this frontend app.
Rules:
- Return backend only in ```javascript server block
- Optionally return schema in ```sql block
- Keep API contracts aligned with frontend fetch usage
- No markdown explanation outside code blocks

这是部署态的后端补全 prompt。它把模型从“开发态先做前端”切换成“发布态补生产后端”。

3. 迭代开发（Iteration Development）

业务链条

编辑 published app
如果你点的是已发布 release，系统不允许直接改 release，而是返回 RELEASE_EDIT_REDIRECT，要求先开 workspace draft。
建立 linked workspace draft
调用 POST /api/apps/:id/ensure-workspace-draft，系统会复制 release 的版本链、文件、文档、必要时连 backend 都一起克隆到 draft。
继续在 draft 中对话
之后所有迭代修改都在 draft / workspace 里完成，调用的仍然是 /api/apps/:id/chat。
保存新版本
每次成功生成都会产生新的 app_versions；preview 也会更新，必要时复用上一版 backend 或新部署 backend。

迭代开发阶段最强的限制

限制影响
MUST_EDIT_FROM_ORIGINAL_VERSION强制基于当前原始版本改，禁止轻易整页重写。
ITERATION_BASELINE要求遵守 APP_SPEC，保留旧能力和兼容性。
API_CONTRACT_BASELINE不能轻易破坏已有 API 结构。
DB_SCHEMA_BASELINE数据库修改应尽量 additive / non-destructive。
allowed change ratio若改动幅度过大，系统会触发一次“最小改动重试”。
workspace early validation如果判断成 destructive rewrite，会直接拦截。

迭代开发限制提示词（基线类）

[MUST_EDIT_FROM_ORIGINAL_VERSION]
You MUST modify based on the original current version code below.
Do not rewrite whole app.
Do not remove existing features unless user explicitly asks.
Keep unchanged sections as-is.
[ITERATION_BASELINE]
Read and follow this APP_SPEC baseline strictly.
Preserve old features/APIs/data compatibility unless explicitly requested.
[API_CONTRACT_BASELINE]
Follow this API contract baseline.
Keep routes compatible and do not remove old APIs unless explicitly requested.
[DB_SCHEMA_BASELINE]
Follow this DB schema baseline.
Use additive, non-destructive changes only.

迭代开发限制提示词（收敛类）

You are updating an existing app. Do MINIMAL CHANGES only.
Rules:
- Keep unchanged code lines exactly as-is whenever possible
- Only edit the smallest necessary sections to satisfy the new request
- Do NOT rename unrelated variables/components
- Do NOT reorder unrelated blocks
- Return complete JSX only in a jsx code block
Restyle the following JSX to match the selected design paradigm.
Rules:
- Keep business logic and API endpoints unchanged
- Rewrite only UI structure/className to satisfy design tokens
- Must include ALL tokens: ...

4. 迭代部署（Iteration Deploy / Draft 覆盖 Release）

业务链条

前提
你编辑的是 linked workspace draft，而不是 release 本体。
保存草稿
点“保存草稿”时只是 POST /api/apps/:id/save-draft，把当前 draft 维持在 status=draft、runtime_mode=local，不会覆盖 release。
部署覆盖
点 deploy 时仍然走 /publish，但 getEffectiveReleaseAppId(appRow) 会把目标 release 找出来。
先做 release backup
如果当前 app 是 workspace draft，发布前会执行 createReleaseBackupSnapshot(releaseAppId, appId)，把 release 当前状态做快照备份。
生成/复用发布后端
系统会判断是否可以复用之前发布的 backend；不行就重新生成 production backend。
健康检查通过后写回 release
发布成功后，不是把 draft 变 published，而是把结果写进 release app 的新版本里。

迭代部署的核心限制

限制影响
不能直接改 release必须先转 workspace draft。
draft / release 分离保存草稿与部署覆盖是两套动作，草稿不会自动覆盖线上。
发布前 backup每次 draft 覆盖 release 前，都会给 release 做备份快照。
可复用发布 backend如果 API/DB 没变，系统会尽量复用上次发布 backend，减少不必要的重建。

5. 回滚（Rollback）

当前代码里的真实状态

我在当前代码里看到了“发布前自动备份 release”的能力，但没有看到一个完整、公开的“回滚到指定 backup” API/UI 链路。

  • 有 app_release_backups 表的写入逻辑。
  • 有 createReleaseBackupSnapshot(releaseAppId, sourceDraftAppId)，会在 draft 覆盖 release 前存：name / icon / description / color / code / server_code / sql_code。
  • 但我没有在当前代码里搜到一个完整的 /rollback 或 restore backup 路由，也没看到前端明确的回滚按钮业务闭环。

现在“已实现”的回滚基础

每次迭代部署覆盖 release 前，系统会给 release 做一次 backup snapshot。
release 本身也会保留 app_versions 历史。
这意味着从“数据基础”上，系统已经具备回滚素材。

现在“未闭环”的部分

没有找到明确的回滚入口 API。
没有看到用户选择某个 backup 后一键恢复 release 的流程。
所以当前更准确的说法是：有备份机制，但回滚产品链路未完整开放。

7. 你最该关注的产品判断

首次生成偏弱点

不是不会生成，而是很容易被“前端优先模式”收窄成先做壳子、先做 JSX。

迭代开发偏保守

当前最明显是“不要重写、尽量兼容、尽量小改”，所以做大重构会有天然摩擦。

部署链条最完整

发布态其实相对成熟：后端补全、DB 校验、Docker 启动、运行时检查、版本写回都已经在了。
