funfo AI Store：革命性迭代计划说明

目标版本基线：0.69.3A ｜ 目的：让发布系统从“复杂、混合、容易歧义失败”收敛为“简单、稳定、可预测、可回滚”。

一句话总结：接下来到底会发生什么变化？

我们不会继续无限堆“自动修复”和“复杂验证”，而是把整个发布系统改造成一个更像正式产品的发布架构：

  • 开发态 和 长期态 彻底分开
  • 新版本先进入 candidate，不是直接冒充已上线版本
  • 只有验证通过，才会晋升成 live
  • 验证失败会 回滚或停止，不再把失败版本继续挂在线上
  • UI 会明确告诉你：现在是 Draft / Candidate / Live / Failed 哪一种

现在的逻辑

复杂
混合
容易歧义
  • 发布时会同时做生成、部署、验证、修复、状态回写
  • 长期化 app 虽然越来越 Docker 化，但历史上混用了 preview 和 live 逻辑
  • 有时 Docker 已启动、页面能打开，但系统仍显示 发布失败
  • 用户需要猜：status + publish_status + app_stage + runtime_mode 才知道现在是什么状态
  • Verifier 阻塞项偏多，容易把“可用但不完美”的 app 判失败

将来的逻辑

简单
单义
可回滚
  • 发布流程改成清晰的 candidate → verify → promote live
  • 失败版本不会污染现网
  • UI 只需要看一个统一状态：release_state
  • Verifier 只阻止真正不能上线的问题
  • 运行状态和发布状态分开表达，不再混淆

核心变化 1：状态模型会怎么变？

现在

系统主要靠下面这些字段组合表达状态：

  • status
  • publish_status
  • app_stage
  • runtime_mode

问题是：用户看不懂，UI 也容易解释错。

将来

统一收敛成一个用户可理解的状态：

  • draft：开发中
  • candidate：正在验证的新发布版本
  • live：已正式上线
  • failed：发布失败，未上线

你以后不需要猜组合状态，只看一个 release_state。

核心变化 2：发布流程会怎么变？

现在： 生成代码 → 启动 Docker → health check → verifier → 失败时可能 runtime 还留着
↓
问题： 可能出现“页面能打开，但系统说发布失败”的矛盾体验
未来： 生成代码 → 部署 candidate runtime
↓
health_check： 只检查这个 candidate 是否真的能跑起来
↓
verifier： 检查这个 candidate 是否满足最低上线标准
↓
通过： promote 为 live
↓
失败： rollback 到上一个 live，或者 stop candidate

核心变化 3：Verifier 会怎么改？

现在的问题

  • Verifier 太像“理想工程审查官”
  • 会检查 artifact、contract、schema、auth smoke、行为等很多东西
  • 容易把“能正常用”的 app 卡在发布门外

未来的变化

  • 只保留真正阻止上线的 blocking checks
  • 把大量“质量建议”降级成 warning
  • 目标不是“完美发布”，而是“稳定发布”

会继续阻止上线的检查

  • runtime 是否能启动
  • /health 是否正常
  • /app/<slug>/ 是否真的可访问
  • prod db mode 是否正确
  • 核心 API 是否存在

会尽量变成 warning 的检查

  • localStorage / mock 风格问题
  • schema 命名和风格问题
  • 产物不够整洁
  • 某些非关键行为 smoke

核心变化 4：开发态和长期态会怎么分离？

开发态

  • Workspace
  • preview-manager
  • 允许快速迭代
  • 允许暂时不完美

长期态

  • Full Docker Runtime
  • 前端+后端同一 app 容器
  • 通过网关 /app/<slug> 访问
  • 按 live/candidate 管理

变化重点

  • preview 不再假装是 live
  • 长期化 app 不再混用 preview 逻辑
  • restart / restore / wake / sleep 都按模式处理

用户会实际感觉到哪些变化？

  • 发布状态会更清楚，不再出现“明明打开了却说失败”的强烈困惑
  • My Apps / Admin 会直接告诉你这是 Draft / Candidate / Live / Failed
  • 失败版本不会继续伪装成上线版本
  • 发布失败原因会更聚焦在真正阻塞上线的问题
  • 成功发布会变得更稳定，而不是靠连续 repair 碰运气

从你的角度，最大的变化是什么？

以前系统更像：尽量把所有事情都自动搞定。

以后系统会更像：把发布这件事做得稳定、单义、能回滚。

这意味着：

  • 少一点“神奇自动修复”
  • 多一点“发布结果明确可预期”
  • 少一点复杂隐藏逻辑
  • 多一点正式发布系统该有的边界

当前执行顺序

  1. R1：状态模型收敛（已开始，先引入统一 release_state）
  2. R2：发布链路收敛（candidate / promote / rollback）
  3. R3：Verifier 降级为最小阻塞模型
  4. R4：开发态 / 长期态彻底分离
