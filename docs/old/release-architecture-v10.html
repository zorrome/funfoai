1 统一状态机（一步到位，不做过渡）

你原方案说"先用派生状态过渡，后续再收敛底层模型"。不要过渡。直接改底层字段。过渡期的双状态并存会制造比改版前更难排查的 bug。

唯一的真相：release_state 字段

状态含义用户看到什么容器行为
draft 仅在 workspace/preview 中存在 "编辑中" preview-manager 临时容器
candidate 已部署候选版本，正在验证 "发布验证中…" candidate 容器启动，但不对外暴露
live 验证通过，正式上线 "运行中" + 绿色徽标 candidate promote → live 容器
failed 验证未通过，候选版本已销毁 "发布失败" + 具体原因 + 操作按钮 candidate 容器停止销毁
rollback 新版本失败，已自动回退到上一个 live 版本 "已回退到 v3" 保留旧 live 容器不动
迁移策略：一次性 migration 把现有 status + publish_status + app_stage + runtime_mode 映射成 release_state，然后删除旧字段。写一个 mapping 函数跑一遍所有 app 记录即可，数据量 ~3000 条，几秒完成。

2 发布管线（candidate → promote/rollback）

这是整个架构的核心杠杆。所有其他改进都是它的推论。

Happy Path

生成 artifact → deploy candidate → health_check → verifier (minimal) → promote → live

Failure Path

生成 artifact → deploy candidate → health_check / verifier → FAIL →
有旧 live → rollback（用户无感） 首次发布 → failed（展示原因 + 操作）

关键规则

  • candidate 不对终端用户可见。只有内部验证流程能访问 candidate 容器。
  • promote 是原子操作：切换路由指向 + 更新 release_state + 记录 live_version_id，三步在一个事务里完成。
  • 旧 live 容器在 promote 后保留 5 分钟再销毁，作为热回退窗口。
  • repair 不在发布管线内。如果 candidate 验证失败，不自动 repair。直接 fail，让用户回到编辑态重新生成。repair 只作为编辑态的辅助手段存在。

3 Verifier Blocking Criteria（具体定义）

你原方案说"只阻止真正不能上线的问题"。这是对的方向，但缺具体 checklist。这里补齐：

🔴 BLOCK（阻止 promote，必须修复）

Bhealth_check 失败（容器无响应或 5xx）
B前端 JSX 运行时崩溃（白屏 / React error boundary 触发）
B前端存在 API 调用但后端无匹配路由（contract mismatch）
BSQL schema 缺失导致后端路由 500
B后端路由返回 shape 与前端期望不匹配（字段缺失/类型错误）
B数据写入后无法正确读回（CRUD 断裂）

🟡 WARNING（不阻止 promote，记录到发布日志）

WCSS 渲染异常（布局偏移、溢出、截断）
W空状态/loading 状态缺失
W非核心 API 路由未实现（次要功能）
W性能问题（响应 >2s 但功能正常）
W日文文案错误/缺失
WConsole warning（非 error）
原则：BLOCK 只管"能不能用"。WARNING 管"好不好用"。用户可以带 warning 上线，后续迭代修复。

4 失败后的用户行动路径

你原方案没有覆盖这个维度。用户看到 failed 之后能做什么，决定了整个体验是"清晰的失败"还是"死胡同"。

场景 A：首次发布失败（没有旧 live 版本）

  • 展示："発布に失敗しました" + 失败原因摘要（来自 verifier BLOCK 项）
  • 操作按钮 ①："編集に戻る" → 回到 workspace 编辑态，带上失败原因作为 context
  • 操作按钮 ②："自動修正を試す" → 触发 repair agent 在编辑态修复，修复后用户手动再次发布
  • 状态：app 回到 draft

场景 B：更新发布失败（有旧 live 版本）

  • 自动行为：rollback 到上一个 live 版本，终端用户无感
  • 展示："新バージョンの発布に失敗しました。v3 に自動ロールバックしました"
  • 操作按钮 ①："新バージョンを編集" → 回到编辑态继续改
  • 操作按钮 ②："失敗ログを見る" → 查看 verifier 详细报告
  • 状态：app 的 release_state 回到 live（旧版本），编辑态有一个 draft 版本可以继续改
核心原则：失败不是死胡同，而是一个带上下文的回退。用户总能从失败状态出发采取下一步行动，且旧版本的在线服务不会中断。

5 Repair 重新定位

你原方案说"减少 repair 次数和权重"。10 分版本更进一步：repair 从发布管线中完全移除，只存在于编辑态。

维度原方案10 分版本
repair 触发时机 发布流程中，verifier 失败后自动 repair 仅在编辑态，用户主动点击"自動修正"触发
repair 上限 减少次数（具体未定） 编辑态最多 2 次自动修复，之后引导用户手动编辑或重新描述需求
发布管线里的 repair 仍保留但减少权重 完全移除。candidate 验证失败 = 直接 fail，不自动补丁
好处 — 发布管线变成纯确定性流程，没有任何 AI 决策环节：deploy → check → pass/fail
为什么这很重要：发布管线内的自动 repair 是原架构中最大的不确定性来源。repair 可能改坏已经正确的部分、引入新 bug、拉长发布耗时。把 repair 移出发布管线后，管线变成一个纯粹的验证+部署流水线，每一步的行为都是可预测的。

6 对 Prompt 系统的影响

架构改版后，你的 prompt 系统需要同步调整。变化其实是简化：

可以删除的 prompt

  • RELEASE_REPAIR_PROMPT — repair 不再存在于发布管线中，这个 prompt 可以完全删除
  • RELEASE_FRONTEND_CONVERSION_PROMPT 中关于"verifier 无法检测 API contract"的防御性措辞 — verifier 只管 BLOCK 项，不再纠结检测方式

需要简化的 prompt

  • RELEASE_ROLE_PROMPT — 不再需要"你的工作不是 self-certify success"这类措辞。生成侧只负责输出 artifact，验证是管线的事。可以收敛为："输出 production-ready 的 frontend + backend + SQL，确保 API contract 对齐。"
  • REPAIR_ROLE_PROMPT — 只需要保留编辑态修复逻辑，删除所有 release 相关的修复策略

需要新增的 prompt

  • Failure Context Prompt — 当用户从发布失败回到编辑态时，把 verifier 的 BLOCK 原因注入到编辑/repair prompt 的 context 中，让 AI 有针对性地修复
