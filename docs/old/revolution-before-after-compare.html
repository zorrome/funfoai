funfo AI Store：革命前后逻辑对照

目的：不做美化，只客观比较“当前逻辑”和“革命后目标逻辑”。

比较结论（一句话）

革命前：生成、部署、验证、修复、上线这几件事耦合在一起，系统状态靠多个字段拼接判断。

革命后：开发、候选发布、正式上线分层处理，状态用统一模型表达，失败版本不再直接暴露成线上版本。

逐项对照

维度 革命前（当前逻辑） 革命后（目标逻辑）
系统目标
  • 同时追求：自动生成、自动修复、自动部署、自动验证
  • 偏向“尽可能自动完成全部流程”
  • 优先目标改为：稳定发布、结果单义、可回滚
  • 偏向“先保证发布正确，再考虑智能修复”
开发态与长期态
  • 开发态与长期态逻辑部分混用
  • preview-manager 与长期 runtime 存在历史耦合
  • 开发态：仅用于 workspace / preview
  • 长期态：仅用于 full Docker runtime
  • 两者职责明确分离
长期化 app 的运行方式
  • 当前已接近 full Docker runtime
  • 但发布、恢复、验证链路里仍有混合痕迹
  • 长期化 app 统一视为独立 Docker 运行单元
  • 前端与后端都由同一个 app 容器承接
发布流程
  • 生成代码
  • 启动 Docker runtime
  • health_check
  • verifier
  • 必要时 repair / re-repair
  • 状态回写

这些步骤强耦合在一个大流程中。
  • 生成代码
  • 部署为 candidate
  • health_check
  • verifier
  • 通过后 promote 为 live
  • 失败则 rollback / stop candidate

上线和验证会被明确分层。
运行状态与发布状态
  • 运行状态与发布状态没有彻底分离
  • 可能出现：runtime 已可用，但 publish 记录为 failed
  • 运行状态单独表达
  • 发布状态单独表达
  • 不会再把“候选运行中”误当成“正式上线”
状态模型
  • 要综合看：status、publish_status、app_stage、runtime_mode
  • 用户和 UI 都需要自己解释组合状态
  • 统一收敛到：draft / candidate / live / failed
  • 先由派生状态过渡，后续再收敛到底层数据模型
Verifier 的角色
  • Verifier 同时承担了“质量审查”和“发布阻断”职责
  • 检查范围较广：artifact、contract、schema、auth smoke、行为等
  • Verifier 改为“最小阻塞守门员”
  • 只阻止真正不能上线的问题
  • 非关键问题降级为 warning
自动修复（repair）
  • 发布流程中允许较多修复与重试
  • 容易拉长链路并增加状态复杂度
  • 减少 repair 次数和权重
  • 优先明确失败，而不是无限补丁
失败版本处理
  • 可能出现失败版本仍然保持容器在线
  • 导致“能打开但显示失败”的体验
  • 失败版本不再作为 live 对外暴露
  • 失败后自动 rollback 或 stop candidate
用户看到的发布结果
  • 可能产生歧义
  • 用户需要自己判断“到底算成功还是失败”
  • 发布结果会更单义
  • 成功就是 live，失败就是 failed，验证中就是 candidate
Admin / My Apps 展示
  • 展示字段较多，但语义分散
  • 对“当前可用性”和“当前发布状态”解释不够清楚
  • 前台与后台都围绕统一状态展示
  • 用户更容易判断一个 app 现在处于哪种发布阶段
系统复杂度
  • 多状态、多阶段、多隐式行为
  • 一旦失败，很难迅速定位是哪一层问题
  • 通过分层和收敛降低复杂度
  • 让失败点更聚焦：生成 / health / verifier / promote / rollback

客观地说，这次革命不是在“增加什么能力”，而是在“删掉什么复杂性”

  • 删除：运行状态与发布状态混在一起的解释方式
  • 删除：失败版本直接暴露成线上版本的风险
  • 删除：过重的 blocking verifier 角色
  • 删除：开发态和长期态之间不清晰的耦合
  • 删除：过多自动修复导致的不可预测性

当前已经开始发生的变化

  • 已经开始引入统一派生状态 release_state
  • 长期化 app 已经在往 full Docker runtime 收敛
  • 接下来真正的关键变化会是：candidate / live 分离
