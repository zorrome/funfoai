AI 调用层：App 迭代 → 迭代发布完成

本文专门说明“已有 live app 的迭代升级”场景下，AI 在哪些阶段被调用、通常会调用几次。

1. 迭代开发阶段的 AI 调用

阶段AI 调用次数说明
在 linked draft 中继续开发/api/apps/:id/chat 主生成至少 1 次/轮根据用户的新需求更新前端，可能附带后端/SQL。
编辑安全收敛incremental / restyle / repair pass可选 0～3 次/轮用于减小破坏性改动、修复前端问题、对齐设计。

2. 迭代发布阶段的 AI 调用

时机AI 调用次数作用
发起 Draft → Candidate / Live → Candidate 升级发布runReleaseManifest(...)通常 1 次从当前 frontend 推导 release 所需结构与路由。
若 frontend 仍偏 local-firstrunFrontendConversion(...)条件触发 0 或 1 次把当前前端改造成可上线的 server-driven 结构。
若后端不可复用runBackendGenerateFromManifest(...)条件触发 0 或 1 次重新生成 backend / SQL。
如果 verifier 失败并进入修复AI repair失败场景下额外触发根据 blocking failure 做定向修复。

3. 当前版本下，迭代升级的 AI 调用量怎么理解

  • 最轻升级：draft 中开发 1 次 AI + 发布 1 次 manifest，后端复用，不需要 conversion。
  • 常规升级：draft 中开发 1～3 次 AI + 发布阶段 2～3 次 AI（manifest / conversion / backend generation）。
  • 复杂失败升级：在常规升级基础上，再追加 AI repair 调用。

4. 关键区别：迭代发布比首次发布更依赖“复用判断”

  • 首次发布：很多东西需要从 0 生成。
  • 迭代发布：系统会先判断后端/SQL 是否可复用。
  • 所以 AI 调用次数在迭代场景下更不固定，也更依赖 diff 情况。

5. 重要结论

  • 迭代开发阶段，AI 更像“持续协作编程器”。
  • 迭代发布阶段，AI 更像“结构分析器 + 转换器 + 后端生成器 + 修复器”。
  • 从成本角度，迭代发布不一定比首次发布更省调用；但如果复用命中高，会显著减少 AI 生成量。
