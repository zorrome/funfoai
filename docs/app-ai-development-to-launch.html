AI 调用层：App 开发 → App 上线

本文只关注“AI 在什么时机被调用、可能调用几次、分别负责什么”。

1. 开发阶段的 AI 调用

时机调用位置调用次数作用
用户在 Workspace 提需求/api/apps/:id/chat 内调用配置好的 LLM provider 流式生成至少 1 次生成主响应，返回前端 JSX，可能附带 backend / SQL。
如果编辑模式下变化过大incremental minimize retry可选 +1 次把过度重写收敛成更小改动。
需要补设计风格 tokenrestyle prompt可选 +1 次只修 UI 风格，不改业务逻辑。
repair passrunRepairPass(...)可选 +1 次对 frontend 进行安全修复。
Phase-B enhancementphase B prompt可选 +1 次在安全前提下增强 UI/交互。

所以：开发阶段 AI 调用次数通常是 1 次，复杂情况下可能到 3～5 次。

2. 首次发布阶段的 AI 调用

阶段调用函数调用次数作用
生成 release manifestrunReleaseManifest(...)1 次从前端产物推导 API / route / release 上下文。
前端转换为 server-drivenrunFrontendConversion(...)条件触发，0 或 1 次若前端呈现 local-first 信号，则转成可发布形态。
生成 backend / SQLrunBackendGenerateFromManifest(...)条件触发，0 或 1 次为 Candidate 生成后端和 SQL。
修复失败时repair / verifier failure prompt失败场景下额外触发针对 blocking failures 做 AI 修复。

也就是说，首次发布不是每次都会调用固定 3 次 AI，而是取决于：

  • 前端是否已可复用
  • 后端是否可复用
  • 是否命中 local-first 转换
  • 是否进入 repair

3. 当前版本下，可以这样理解 AI 调用总量

  • 最轻路径 开发 1 次 AI + 发布 1 次 AI（manifest）+ 可能 0 次 backend 生成（复用）。
  • 常规路径 开发 1～3 次 AI + 发布 2～3 次 AI（manifest / conversion / backend）。
  • 复杂失败路径 在常规路径基础上，再进入 AI repair，多一轮或多轮修复调用。

4. 重要结论

  • 开发阶段 AI 调用更偏向“生成与局部修复”。
  • 发布阶段 AI 调用更偏向“结构化产物提取、转换、后端补全、失败修复”。
  • 当前系统不是“一次 AI 包办所有”，而是“主生成 + 条件化补充调用”的链式结构。
